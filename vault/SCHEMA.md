# SCHEMA — FIELDBRAIN 볼트 운영 규칙

> 이 파일은 로컬 LLM이 매 작업 전에 읽는 **운영 매뉴얼**이다. 보고 양식·갱신 규칙·질의 규칙·lint 규칙을 정의한다.
> **모든 데이터는 가상 훈련 시나리오다. 실제 부대·지명·좌표·교범과 무관하며, 공개·합성 데이터만 사용한다.**
> 표기 규약: 부대명(엑스레이대대 등)·교범명(경계교범(가상) 등)·좌표(TRN 격자)는 **실존 대상과 무관한 창작 표식**이다.

---

## 0. 컴플라이언스 (최우선)

- 이 볼트의 모든 로그·보고·교리·좌표·부대명은 **가상(synthetic)**이다. `exercise: true` 표식을 모든 raw 문서 frontmatter에 붙인다.
- **본문 첫 줄에 훈련 배너를 필수로 넣는다**(§2). 단일 문서가 볼트 밖으로 유출·스크린샷돼도 가상임이 드러나야 한다.
- 군사기밀·대외비·실제 작전정보 **일절 금지**. 실제 교범 번호·실존 부대 별칭·실측 좌표를 쓰지 않는다.
- 답변에 근거가 없으면 **지어내지 말고 거부**한다(§6).

---

## 1. 볼트 구조 (3층)

```
fieldbrain-vault/
├── SCHEMA.md              # 이 파일 — 운영 규칙 (LLM 읽기전용)
├── raw/                   # 불변 · write-once · LLM 읽기전용
│   ├── doctrine/          # 교리·교범 발췌 (가상 예시)
│   ├── logs/              # 상황일지 (이벤트 직렬화)
│   └── comms/             # 수신 보고 원문 (SALUTE 등)
└── wiki/                  # LLM 생성·유지보수
    ├── situation.md       # 현재 상황판 (METT-TC)
    ├── units/             # 부대 엔티티 페이지
    └── lessons/           # 전훈
```

- **raw/** 는 절대 수정하지 않는다. 정정이 필요하면 새 로그를 추가하고 이전 로그를 `supersedes`로 참조한다.
- **wiki/** 는 LLM만 쓴다. 모든 문장은 raw 근거로 소급 가능해야 한다.

## 2. 보고 양식 (raw 문서)

모든 raw 문서는 (1) YAML frontmatter → (2) 훈련 배너 한 줄 → (3) 본문 순으로 작성한다.

```yaml
---
id: LOG-0703-1432-02CO        # <종류>-<MMDD>-<HHMM>-<출처> 고유 ID
type: 상황일지                 # 상황일지 | 최초보고 | 중간보고 | 작전개요 | 작전변경 | 기상 | 교리
report_ts: 2026-07-03T14:32:00+09:00   # ISO 8601, 정렬·최신판단 기준
unit: 2중대                    # 보고 부대
reporter: 2중대 상황병
clearance: 1                   # RBAC 등급 (§4)
location_grid: TRN 1615 6830   # 가상 훈련 좌표 (§2 좌표 규약)
entities: [대항군-정찰조, 마령고지] # 위키 크로스링크 대상
source_channel: 무전→상황병 전기
exercise: true                 # 훈련 데이터 표식 (필수)
supersedes: LOG-0703-1040-BN   # (선택) 이 보고가 무효화하는 이전 보고
refs: [LOG-0702-0634-02CO]     # (선택) 관련 보고
---

> **[훈련·가상]** FIELDBRAIN 데모용 합성 시나리오. 실제 부대·작전·좌표·교범과 무관.

(본문)
```

### 파일명 규약
파일명은 사람 편의용이며 `<종류>-<MMDD>-<HHMM>-<설명>.md`(예: `LOG-0703-1900-작전변경.md`) 형식을 권장한다.
**문서의 정본 id는 오직 frontmatter의 `id`이며, 파서는 파일명에서 id를 추론하지 않는다.** 파일명과 id가 달라도 무방하다.

### 좌표 규약 (중요)
- `location_grid`는 **`TRN <4자리> <4자리>`** 형식의 **가상 사판(sand-table) 상대좌표**다. 가상 원점 기준 상대값이며 **실측 MGRS·실지형과 무관**하다.
- `TRN`은 실존 MGRS 100km 격자식별자(2문자)가 아니므로 실좌표로 역산되지 않는다. 좌표 간 상대 거리(방위·이격)만 의미를 갖는다.

### type별 용도
| type | 의미 | 예시 |
|---|---|---|
| 상황일지 | 관측·경계·행정 등 일상 이벤트 | 드론 관측, 병력 이동 |
| 최초보고 | 접촉·교전·사건 최초 발생 | 소규모 접촉 |
| 중간보고 | 진행 중 상황 갱신 | 병력·탄약 현황 |
| 작전개요 | 작전명령 요약(OPORD) | 방어준비·보급로 지정 |
| 작전변경 | 기존 명령 변경(FRAGO) | 보급로 우회 하달 |
| 기상 | 기상·수문 관측 | 강우량·하천 수위 |
| 교리 | 교범·SOP 발췌 (가상) | 경계작전 절차 |

## 3. ingest 갱신 규칙 (상황병)

신규 raw 문서 1건이 저장되면:

1. **파일 감시** → 변경분만 재청킹·재임베딩(증분). 삭제/이동 시 해당 청크 제거.
2. **엔티티 추출**: frontmatter `entities` + 본문에서 부대·위치·시각·장비를 뽑아 `wiki/units/<엔티티>.md`에 관측 1건 누적(크로스링크 `[[엔티티]]`).
3. **상황판 반영**: `wiki/situation.md`의 해당 METT-TC 항목을 갱신하되, 반드시 근거 로그 ID를 인라인 인용한다.
4. **원본 불변**: raw 문서 자체는 절대 고치지 않는다.

## 4. RBAC 등급 (clearance) — 가상 기준

| 등급 | 명칭(가상) | 접근 역할 | 예시 |
|---|---|---|---|
| 1 | 일반 | 소대장 이상 전원 | 일상 관측·경계·기상 보고 |
| 2 | 대외비급(가상) | 대대 참모 이상 | 작전개요·작전변경·병력배치 · **적 접촉/교전 최초보고 · 상세 첩보(SALUTE)** |
| 3 | 제한(가상) | 대대장·연대 참모 | 상급 지휘관 의도, 민감 첩보 |

- **등급 부여 원칙**: 일상 관측(드론·경계)은 등급 1. **작전명령(작전개요·작전변경)과 적 접촉·교전·상세 첩보는 첩보 가치가 높아 등급 2**를 기본으로 한다.
- 검색은 **벡터 유사도 계산 이전에** `clearance <= 사용자등급` 프리필터를 적용한다(권한 밖 문서는 컨텍스트 진입 자체가 불가).
- 필터로 제외된 문서 건수를 UI에 `접근 차단 N건`으로 표기한다.

## 5. 데모 시나리오 매핑 (어느 로그가 어느 장면을 지원하는가)

> 아래 id는 표시용 축약이 아닌 **전체 id**다. 인용 시에도 전체 id를 그대로 쓴다(§6).

| 데모 장면 | 근거 로그(전체 id) |
|---|---|
| 드론 반복 정찰 패턴 | LOG-0702-0634-02CO · LOG-0703-0629-02CO · LOG-0704-0631-02CO (3회, 06:30±5, 은하능선 동일 축선) |
| 모순 보고 → lint | LOG-0703-1432-02CO ↔ LOG-0703-1432-RCN (동일 시각, 상반된 대항군 정찰조 위치) |
| 작전변경 최신 우선 | OPORD-0703-1030-BN(원본 작전개요) → LOG-0703-1040-BN(요약) → LOG-0703-1900-BN(작전변경, supersedes) |
| 대대 방어 OPORD (5개항) | OPORD-0703-1030-BN + 적전술판단 DOC-적전술-01 |
| 좌표×기상 도하 판단 | LOG-0704-0800-WX(수심·유속) + 도하교범(가상) + LOG-0703-1900-BN(백석교 통제) |
| RBAC | clearance 2 문서(작전개요·작전변경·교전 최초보고·SALUTE·중간보고) 접근 여부 |
| 인수인계·상급보고 요약 | 07-03~07-04 전 로그 시간순 종합 |

## 6. query 규칙 (작전참모)

1. **인용 강제**: 모든 사실 문장에 근거를 붙인다.
   - 교리: `[교범명(가상) §조항]` — 예: `[경계교범(가상) §2.3]`
   - 보고: `[<type> <전체 id> · MM-DD HHMM]` — 예: `[상황일지 LOG-0703-1432-02CO · 07-03 1432]`
2. **인용 검증**: 답변의 인용이 실제 검색된 청크에 존재하는지 사후 검증한다. **인용 id는 반드시 frontmatter의 전체 id와 문자열 일치**해야 하며, 불일치 인용은 삭제/재생성한다.
3. **최신 우선 중재**: 보고끼리 상충하면 `report_ts`가 최신인 보고가 유효하다. 교리는 일반 원칙이며 최신 보고가 현재 상황을 우선한다. **상충 사실은 숨기지 말고 명시**한다.
4. **거부 조건**: 검색 최상위 유사도가 임계값 미만이거나 RBAC 필터로 근거가 비면, LLM 호출 없이 `"교범·보고에서 근거를 찾을 수 없습니다"`로 거부한다.
5. **포지셔닝**: 참모 보좌·상황 기억용이다. 표적·사격 결심을 대신하지 않는다.

## 7. lint 규칙 (정보장교) — 첩보 교차검증

원본 로그는 각 부대가 **독립적으로 관측·상신한 순수 청보**다. 모순 교차검증은 원본이 아니라 **이 lint 단계에서 상급(정보/작전)이 사후 수행**한다.

상시 점검하여 `⚠ 재확인 요구`를 상황판에 플래그한다.

1. **위치 모순**: 동일 엔티티(예: `대항군-정찰조`)에 대해 근접 시각(±30분)의 상반된 위치 보고가 있으면 플래그.
2. **스테일**: 상황판 항목의 최신 근거 로그가 기준 시간(예: 6시간) 이전이면 `갱신 필요` 플래그.
3. **고아**: 어떤 wiki 문장도 근거 raw로 소급되지 않으면 플래그.
4. **상충 명령**: `작전개요`와 `작전변경`이 동일 대상(예: 보급로)에 상반되면, 최신을 유효로 하되 이전 명령의 잔존 참조를 플래그.

lint는 결론을 내리지 않는다 — **재확인을 요구**할 뿐, 판단은 지휘관 몫이다.

## 8. 인용 표기 예시

> 최신 보고 기준 청류천 여울목 실측 수심이 도보 도섭 안전 한계에 도달했습니다 `[기상 LOG-0704-0800-WX · 07-04 0800]`. 유속 증가로 도섭 기준상 재검토가 필요합니다 `[도하교범(가상) §4.1]`. 단, 지정 우회 도섭지는 백석교 통제에 따른 것입니다 `[작전변경 LOG-0703-1900-BN · 07-03 1900]`.
> ⚠ 재확인 요구: 지정 우회 도섭지(여울목)가 기상으로 위험 — 대체 도하수단 검토 필요.
